# 이게 뭐야? (3분)

## 한 문장으로

**한 명이 여러 개의 모바일 앱을 동시에 운영하기 위한 Spring Boot 백엔드 템플릿 레포입니다.**

## 조금 더 풀어서

모바일 앱을 여러 개 만들고 싶은데, 각 앱마다 백엔드를 따로 만들면 관리가 불가능해지죠. 이 레포는 **한 번 셋업해두면 새 앱을 추가할 때 셋업 비용이 거의 0 에 가까운** 구조를 제공합니다.

비유하자면:

> **공장 하나, 제품 여러 개**

가계부 앱 · 운동 기록 앱 · 포트폴리오 트래커 앱 — 세 앱이 전부 다른 도메인이지만, 공통으로 필요한 건 똑같아요. 이메일 가입 · 소셜 로그인 · 푸시 알림 · 파일 업로드 · DB 분리. 이 공통 부분을 **한 JAR 안에** 묶어두고, 각 앱의 도메인만 따로 꽂는 구조입니다.

## 어떤 문제를 풀었나

### 문제 1 — 혼자서 앱 여러 개 운영하면 결국 뭔가를 포기하게 됨

평범한 패턴:

- 앱 A 서버 + 앱 B 서버 + 앱 C 서버 → **서버 3대를 모니터링, 업데이트, 보안 패치** → 1 명이 감당 불가
- 마이크로서비스로 쪼갠다 → **쿠버네티스 · 서비스 메시 · 분산 추적** → 1 명이 감당 불가
- "앱 A 만 해야겠다" → **앱 공장 전략 자체 포기**

이 템플릿의 해법: **한 JVM 프로세스 안에서 여러 앱 도메인이 공존** 하는 모듈러 모놀리스. 서버는 1 대, 앱은 N 개.

### 문제 2 — 새 앱 추가하면 똑같은 셋업을 반복하게 됨

"이메일 인증 로직" · "JWT 토큰 발급" · "소셜 로그인" 은 앱마다 거의 동일. 하지만 매번 복붙하면 나중에 수정할 때 N 곳을 고쳐야 함.

이 템플릿의 해법: **`core/` 모듈** 로 공통 로직을 추출 + **앱별 schema 분리** 로 데이터는 완전 독립 + **`./tools/new-app/new-app.sh sumtally` 한 줄** 로 새 앱 스캐폴드 전체 자동 생성.

### 문제 3 — 템플릿을 복사한 후 원본 업데이트를 못 받음

평범한 fork 방식: "원본에 좋은 기능이 추가됐는데 내 레포로 가져오려면 rebase 지옥."

이 템플릿의 해법: **GitHub Template Repository 패턴** + **Conventional Commits 강제** → `git log --grep="^feat\\|^fix"` 로 "공통 개선 커밋만" 기계 필터링 → 파생 레포로 cherry-pick.

## 당신한테 맞을 수 있는 경우

✅ 모바일 앱을 **혼자 또는 2~3 명** 규모로 여러 개 만들고 싶다  
✅ 각 앱이 **서로 다른 도메인** 이지만 인증 · 푸시 · 결제 같은 **공통 기능은 재사용** 하고 싶다  
✅ **Spring Boot 에 이미 익숙** 하거나 배우고 싶다  
✅ 관리형 서비스 (Supabase · Firebase · Resend 등) 결합으로 **운영 부담을 최소화** 하고 싶다  
✅ **PostgreSQL** 을 DB 로 쓰는 게 괜찮다

## 당신한테 안 맞을 수 있는 경우

❌ 팀이 **10 명 이상** 이고 앱마다 전담자가 있다 → 마이크로서비스 / 별도 레포가 더 적합  
❌ **Node.js · Go · Python** 등 다른 언어 생태계를 원한다 → 이 레포는 Java 전용  
❌ **MongoDB · DynamoDB · Firestore** 등 비관계형 DB 가 필수다 → PostgreSQL 전제로 설계됨  
❌ 앱이 **하나뿐** 이고 당분간 추가 계획도 없다 → 과한 설계. 일반 Spring Boot 튜토리얼이 나음  
❌ **HA 99.99% · 멀티 리전 · 초당 10,000 요청** 급 스케일이 필요하다 → 이 템플릿은 인디 스케일 전제 ([ADR-007](../philosophy/adr-007-solo-friendly-operations.md))

## 숫자로 보기

| 항목 | 값 | 의미 |
|---|---|---|
| 앱 추가 시간 | 약 **1~5 분** | `new-app.sh <slug>` 한 줄 |
| 공통 코드 재사용률 | 거의 **100%** | 인증 · 푸시 · 스토리지 · 이메일 전부 공유 |
| 배포 단위 | **JAR 1 개** | 앱이 10 개여도 배포는 한 번 |
| 외부 월 비용 (인디 스케일) | **$50 ~ $100** | Supabase + Resend + FCM + 도메인 + 호스팅 |
| ADR 문서 수 | **16 개** | 주요 설계 결정의 근거 기록 |
| ArchUnit 규칙 수 | **22 개** | 경계 위반은 빌드 시 실패 |

## 어떤 기술을 쓰나 (훑기)

| 영역 | 기술 |
|---|---|
| 언어 · 프레임워크 | Java 21, Spring Boot 3.3 |
| 빌드 | Gradle 멀티모듈 |
| DB | PostgreSQL 16 (Supabase 또는 로컬 Docker) |
| 인증 | JWT (HS256) + Spring Security |
| 이메일 | Resend |
| 푸시 | Firebase Cloud Messaging |
| 스토리지 | MinIO (S3 호환) 또는 Cloudflare R2 |
| 배포 | Docker + Kamal + blue/green · 맥미니 홈서버 또는 클라우드 VM |
| 관측성 | Prometheus + Loki + Grafana |
