# Documentation Style Guide

> **유형**: Reference · **독자**: Level 2~3 (저자) · **읽는 시간**: ~15분

이 문서는 **어떻게 문서를 쓸 것인가** 의 규칙을 모읍니다. 추상 원칙이 아니라 **`philosophy/` 의 36 개 ADR 파일럿에서 귀납적으로 추출된 규칙** 입니다 (초기 36 개 ADR 파일럿 시점에 추출, 이후 동일 규칙 적용). 각 규칙마다 구체 샘플 링크가 있어야 하며, 샘플 없는 규칙은 근거가 약합니다.

> **이 문서의 위상**: `docs/` 안의 모든 문서가 참조하는 메타 문서. 새 문서 쓸 때 이 가이드를 따르고, 예외가 필요하면 "왜 이 경우에 예외인가" 를 본 문서에 추가.

## 1. 독자 페르소나 — Level 0~3

모든 문서는 **어느 레벨의 독자를 대상으로 쓰는지** 를 명시적으로 결정해야 합니다.

| Level | 누구 | 배경 지식 | 예시 문서 타겟 |
|---|---|---|---|
| **Level 0** | 처음 코드를 짜보는 사람 | 프로그래밍 기초만 | "프로젝트 소개", "첫 앱 실행" (아직 미작성) |
| **Level 1** | 주니어 백엔드 개발자 | Spring Boot 튜토리얼 수준 | `start/onboarding.md`, `start/app-scaffolding.md` |
| **Level 2** | Spring 실무 중급 | 2+년 현업, 운영 경험 | `structure/architecture.md`, 대부분의 `convention/*.md` |
| **Level 3** | 시스템 설계 관심자 | 여러 언어/스택 경험, 트레이드오프 사고 | `philosophy/*.md` 전체 |

### 레벨별 톤 규칙

**Level 0~1** — 비유 · 예시 · 단계별 안내 위주:
- 긴 개념 설명 전에 "이게 뭐에 쓰이는가" 먼저
- 코드 스니펫은 **전체 맥락** 을 담아서 (import 포함)
- 실행 명령은 그대로 복붙 가능한 형태

**Level 2** — 의사결정 맥락 + 구체 구현:
- "왜" 와 "어떻게" 를 같이 다룸
- 코드 스니펫은 **핵심만 발췌** 가능 (import 생략 허용)
- 업계 표준과의 비교 자연스럽게

**Level 3** — 트레이드오프 · 대안 검토 · 장기 결과:
- "왜" 중심으로 다루고, "어떻게" 는 Code References 링크로 위임해요
- 여러 대안을 병렬 비교 (Option 1/2/3 구조)
- 미래 부작용 · 이행 경로까지 기록

### 샘플

- Level 3 예시: [`ADR-005 · 단일 Postgres database + 앱당 schema`](../philosophy/adr-005-db-schema-isolation.md) — 대안 3개 비교 + 5중 방어선 · 미래 이전 경로
- Level 2 예시: [`Design Principles`](../convention/design-principles.md) — SOLID 설명 + 코드 샘플
- Level 1 예시: [App Scaffolding (`new-app.sh`)](../start/app-scaffolding.md) — 단계별 명령 + 트러블슈팅

## 2. 문서 유형 5 종 + 공통 메타블록

모든 문서에 **ADR 규격을 강제하지 않습니다**. ADR 은 "결정의 근거" 를 담는 특수 형식이고, 다른 성격의 문서 (참조표 · 절차서 · 설명글 · 운영 런북) 는 각자에 맞는 규격이 따로 있습니다.

### 2.0 공통 메타블록 (전 문서 공통)

h1 바로 아래에 **한 줄 블록** 필수:

```markdown
# {문서 제목}

> **유형**: {ADR | Reference | How-to | Explanation | Runbook} · **독자**: Level {0~3} · **읽는 시간**: {N분}
```

독자가 클릭하자마자 "이건 어떤 성격의 문서인가" 를 즉시 인지할 수 있어요. 유형별 규격 차이가 **혼란이 아니라 목적에 맞는 구조** 로 받아들여져요.

**예외 — README / 인덱스 진입점**:
- `docs/README.md` (GitHub 진입점) 은 메타블록 생략이 가능해요. 사용자의 첫 진입 동선이라 *유형 라벨* 보다 *바로 인덱스* 가 자연스러워요
- `docs/{group}/README.md` (그룹 인덱스) 는 메타블록 권장 (인덱스 자체도 *Reference* 유형)
- 예외 적용 시 *왜 메타블록이 없는지* 한 줄 주석으로 명시 권장

### 2.A — ADR (Architecture Decision Record)

**대상**: `philosophy/` 의 설계 결정 카드. 17 파일.
**독자**: Level 3 (시스템 설계 관심자).
**핵심 가치**: "왜 이 선택을 했는가 + 대안 대비 근거 + 장기 결과".

`philosophy/` 의 36 ADR 은 아래 **8 개 필수 섹션** + 선택 섹션으로 구성. 새 설계 결정을 ADR 로 기록할 때 그대로 따름.

### 필수 섹션 순서 (h2)

1. **결론부터** — 30초 안에 핵심 잡기. 4~6 줄.
2. **왜 이런 고민이 시작됐나?** — 문제 맥락. 마지막에 `> 이 결정이 답할 물음은 이거예요` 블록으로 물음 명확화.
3. **고민했던 대안들** — 최소 2~3 개. 각 대안은 `### Option N — 제목` 서브섹션. 채택안은 `★ (채택)` 표시.
4. **결정** — 실제 채택된 구현이에요. 구조도 / 코드 스니펫 / 하위 결정을 포함합니다.
5. **이 선택이 가져온 것** — `### 긍정적 결과`, `### 부정적 결과` 서브섹션 필수. 정직하게.
6. **교훈** — 3~4 개 정도. 각 교훈은 `### 교훈 제목` 서브섹션 + 구체 일화 + `**교훈**:` 한 줄 요약.
7. **관련 사례 (Prior Art)** — 업계 레퍼런스 5~7 개 외부 링크.
8. **Code References** — 실제 파일/라인 링크. GitHub raw URL 사용.

### 축약형 ADR — 대안 부재 사유 노트 의무

ADR 의 결정이 **한 방향적** 이라 (예: 도메인 분리 / 모듈 추출) *진짜 대안 비교* 가 무의미한 경우, 섹션 3 (고민했던 대안들) 을 *축약 / 생략* 가능. 단:

- **축약 사유 노트 필수** — 섹션 3 진입부에 `> **대안 분석의 한계**` 또는 `> **이 ADR 은 *한 방향적 결정*** ` 같이 *왜 정형 대안 비교가 부적합한지* 한 단락 명시
- **고려한 형태 짧게** — *진짜 대안 X* 라도 *고려한 형태* (예: "auth 안에 두기", "별도 microservice") 를 짧게 기록. reader 가 *왜 이 형태가 부적합한가* 이해 가능
- **다른 7 섹션은 보유 의무** — 결론부터 / 왜 / 결정 / 이 선택이 가져온 것 / 교훈 / Prior Art / Code References 는 그대로

참조: [`ADR-024`](../philosophy/adr-024-email-domain-extraction.md) — 도메인 추출의 한 방향적 결정 + 대안 부재 사유 노트 사례

### 파일 헤더 (h1 + Status 줄)

```markdown
# ADR-NNN · 제목 한 줄

**Status**: Accepted/Deprecated/Superseded. YYYY-MM-DD 기준 현황 한 줄.
```

- h1 제목 포맷: `ADR-NNN · 제목` (중점 ` · ` 구분자)
- Status 줄은 **파일 상단 고정**. 날짜 · 현재 유효성 · 주요 구현 위치 한 줄 언급.

### 선택 섹션

- `## 비목표` — 명시적 거절이 중요할 때 ([`adr-007`](../philosophy/adr-007-solo-friendly-operations.md) 참조)
- 결정 안의 `### 설계 선택 포인트` — 여러 하위 결정이 있을 때 ([`adr-009`](../philosophy/adr-009-base-entity.md) 의 포인트 1~5)

### 샘플

- 전체 구조 모범: [`ADR-009 · BaseEntity 공통 슈퍼클래스`](../philosophy/adr-009-base-entity.md) — 8 섹션 + 설계 포인트 5개
- "비목표" 추가 예: [`ADR-007 · 솔로 친화적 운영`](../philosophy/adr-007-solo-friendly-operations.md)

### 2.B — Reference (참조 문서)

**대상**: 규칙 · 목록 · 스펙 · 표. 찾아보기 용.
**독자**: Level 2 (개발 중 규약 확인).
**핵심 가치**: "어떤 규칙/항목이 있는가 → 빠르게 찾기".

**필수 섹션 순서 (h2)**:

1. **개요** — 한두 줄. "이 문서가 담는 항목 = 무엇". 전체 독자가 10초 안에 판단.
2. **(본문)** — 표 · 목록 · 카테고리별 항목. anchor 링크로 즉시 점프 가능하게.
3. **관련 문서** — 이 규약을 적용하는 구체 가이드 / 이 규칙의 근거 ADR.

**권장 요소**:

- 각 항목마다 **Good / Bad 예시** 쌍 (반례가 있어야 규칙이 명확)
- 의사결정 트리 (조건별 어느 옵션을 고를지)
- 버전/날짜 (규칙이 바뀌었을 때 추적)

**피해야 할 것**:

- "왜 이 규칙인가" 장황하게 설명 — 이건 ADR 로 옮겨야 해요. Reference 는 **규칙 자체** 에 집중해요.
- 절차 (Step 1/2/3) — 이건 How-to 예요. Reference 는 정적 목록이에요.

**모범**:

- [`Naming Conventions`](../convention/naming.md) — 네이밍 규칙 카테고리별 표 + 반례
- [`Architecture Rules (ArchUnit)`](../structure/architecture-rules.md) — ArchUnit r1~r22 명세

### 2.C — How-to (절차서)

**대상**: 특정 목표 달성 단계별 가이드.
**독자**: Level 1~2 (실행자).
**핵심 가치**: "이걸 따라하면 X 가 완성됨".

**필수 섹션 순서 (h2)**:

1. **개요** — 목표 · 대상 독자 · 예상 시간 · 사전 요구사항.
2. **단계 1 ~ N** — 각 단계에 (1) 무엇을, (2) 명령 / 코드, (3) 성공 검증, (4) 실패 시 분기를 포함합니다.
3. **트러블슈팅** — 자주 만나는 에러 + 해결법.
4. **다음 단계** — 이 가이드 다음에 할 것.

**권장 요소**:

- "한눈에 보기" ASCII 다이어그램 (전체 플로우) — 상단에.
- 명령어는 **복사-붙여넣기 가능** 형태 (주석 포함 OK, 그대로 실행됨).

**피해야 할 것**:

- 왜 이 방법인가 — ADR 또는 Explanation 에 위임해요.
- 개념 설명이 너무 길어짐 — How-to 는 "따라가면 되는" 체계여야 해요.

**모범**:

- [`첫 수정 — nickname 컬럼 추가`](../onboarding/first-change.md) — nickname 컬럼 추가 튜토리얼
- [`키 교체 절차 (Key Rotation)`](../production/setup/key-rotation.md) — 키 교체 절차

### 2.D — Explanation (설명글)

**대상**: 개념 · 구조 · 동작 원리 설명.
**독자**: Level 2~3 (이해하고 싶은 사람).
**핵심 가치**: "이 시스템이 어떻게 생겼고 왜 이렇게 동작하는가".

**필수 섹션 순서 (h2)**:

1. **한 문장 요약** — 이 설명의 핵심을 한 문장으로.
2. **개념** — 원리 · 배경. **다이어그램 필수** (ASCII 또는 그림). 말로만 설명하면 Explanation 자격 미달.
3. **구조** — 컴포넌트 · 각자의 책임 · 상호작용. 표 권장.
4. **사용법 / 예시** — Good example + Bad example 최소 1쌍.
5. **관련 문서** — 이 개념이 전제하는 ADR, 이 개념을 사용하는 How-to.

**권장 요소**:

- FAQ (흔한 오해)
- 확장 시나리오 (이 개념을 더 밀어붙이면?)

**피해야 할 것**:

- 단계별 절차 — How-to 에 속해요. Explanation 은 정적 이해를 다뤄요.
- 결정 근거만 — ADR 에 속해요. Explanation 은 "현재 상태" 를 설명합니다.

**모범**:

- [`JWT Authentication`](../structure/jwt-authentication.md) — JWT 시스템 구조 + 필터 체인
- [`Multi-tenant Architecture`](../structure/multitenant-architecture.md) — per-app schema 구현 패턴

### 2.E — Runbook (운영 절차)

**대상**: 장애 대응 · 정기 운영 작업.
**독자**: Level 2.5 (운영자).
**핵심 가치**: "지금 이 상황이면 뭘 해야 하나" 를 30초 안에.

**필수 섹션 순서 (h2)**:

1. **개요** — 이 runbook 이 다루는 상황 범위.
2. **상황 매트릭스** — 증상 → 원인 → 대응 → 참조 섹션의 표. 빠른 점프용.
3. **(각 상황별 상세 섹션)** — 진단 명령 · 대응 명령 · 확인 명령. **전부 복사-붙여넣기 가능**.
4. **에스컬레이션** — 이 runbook 으로 해결 안 될 때.

**권장 요소**:

- 체크리스트 (대응 완료 확인용)
- 사후 분석 링크 (이 장애 발생 후 무엇을 문서화/백로그에 추가할지)

**피해야 할 것**:

- 장황한 배경 설명 — 긴급 상황에서 방해.
- 개념 · 결정 근거 — ADR / Explanation 에 위임해요.

**모범**:

- [`운영 런북 (Runbook)`](../production/deploy/runbook.md) — 평시 배포 · 롤백 · 장애 대응

### 2.F — 유형 선택 가이드

새 문서를 쓰기 시작할 때 **이 질문 순서** 로 유형 결정:

1. **"왜 이 선택을 했는가?"** 를 설명하는가 → **ADR**
2. **"어떤 규칙/항목이 있는가?"** 를 정리하는가 → **Reference**
3. **"이걸 하려면 어떻게?"** 단계별 절차인가 → **How-to**
4. **"이 시스템이 어떻게 생겼는가?"** 개념/구조 이해인가 → **Explanation**
5. **"이 상황이 터지면 뭘 해야 하나?"** 긴급 대응인가 → **Runbook**

**복합 유형 허용**: 한 문서가 Explanation + Reference 성격을 동시에 가질 수 있음 (예: `testing-strategy.md` 는 4층 전략 설명 + 각 층 체크리스트). 메타블록에 **주 유형** 만 명시, 본문에서 섞어도 OK.

## 3. 톤 · 문체 · 말투

### 기본 말투: 해요체 + 합니다체 병행

- **친근한 흐름 / 설명 / 사용자 직접 말 걸기 / 비유** — 해요체 (`~예요`, `~이에요`, `~해요`, `~돼요`)
  - 예: "이 문서는 ... 정리해요", "막히면 ... 보세요", "감이 잡혀요"
- **규칙 / 정의 / 단정 / 학술적 진술** — 합니다체 (`~합니다`, `~따릅니다`, `~됩니다`, `~깨집니다`)
  - 예: "이 클래스는 한 가지 책임만 가집니다", "Hibernate 가 자동으로 prefix 를 붙입니다", "Port 가 엔티티를 노출하면 추출이 깨집니다"
- **명령조** (`~하라`, `~할 것`) 은 금지. 사용자에게 행동 권유 시엔 `~하세요`.
- **학술 톤** (`~하여야 한다`, `~이며`, `~함`, `~됨`) 은 회피.

판단 기준 — *교과서적 진술 / 코드 동작 정의 / 규칙 서술* 은 합니다체, *독자에게 풀어서 설명하는 흐름* 은 해요체. 둘이 자연스럽게 병행되어야 부드러움 과잉이 되지 않습니다.

### 컨텍스트별 톤 분기 매트릭스

문서 안에서 *어느 자리* 에 *어느 톤* 이 어울리는지 명시. 일률적 해요체는 *반말 같은 인상* 을 주고, 일률적 합니다체는 *교과서 톤* 으로 무거워집니다.

| 컨텍스트 | 권장 톤 | 예시 |
|---|---|---|
| **Status 줄** (ADR 헤더) | 합쇼체 | "Accepted. 2026-05-01 기준 ... 분리했어요" → "**~로 분리했습니다**" |
| **결론부터** (요약 4~6줄) | 병행 | 핵심 사실 = 합쇼체, 부연 설명 = 해요체 |
| **배경 / 왜 이런 고민이** | 해요체 위주 | 독자에게 *왜 이게 문제인지* 풀어 설명 → "운영 시:", "...막혀요" |
| **결정 표 / 정책 항목** | 합쇼체 | "이 클래스는 한 가지 책임만 **가집니다**", "Hibernate 가 prefix 를 자동으로 **붙입니다**" |
| **대안 비교 (장점 / 단점 / 탈락 이유)** | 병행 | "단순해요 / 컴파일 보장이 **없어요**" — bullet 의 짧은 평가는 해요체, *원리 단정* 은 합쇼체 |
| **이 선택이 가져온 것 (긍정 / 부정)** | 합쇼체 위주 | 객관적 결과 진술 → "추출이 1분 컷이 **됩니다**", "키 유출 시 전면 위조가 **가능합니다**" |
| **교훈** | 해요체 위주 | 대화체 회고 → "...라는 걸 6개월 후 깨달았어요", "교훈: ~해야 해요" |
| **비목표 / 안 다루는 범위** | 합쇼체 | 정책 선언 → "고가용성 99.99% SLA 는 **추구하지 않습니다**" |
| **검증 결과 / 검증 명령** | 합쇼체 | 객관적 보고 → "5/5 PASS 로 **확인됩니다**", "build PASS 로 **검증했습니다**" |
| **트러블슈팅 / FAQ** | 해요체 | 독자 직접 말 걸기 → "...가 안 보이면 ...를 확인하세요", "이 경우 ...해요" |
| **bullet 안 짧은 라벨** (`- **결정**: ...`) | 명사구 또는 합쇼체 | "결정: 포트원 채택" / "결정: 포트원으로 **결정합니다**" — 둘 다 OK, 일관 유지가 핵심 |
| **헤딩 / 섹션 제목** | 명사구 | "## 결정 — 3 도메인 분리" / "### 옵션 B — Hybrid 정책" |
| **ASCII 다이어그램 / 표 셀** | 명사구 | 짧은 라벨이라 명사구 자연 |

**금지** (어디서든):
- 음슴체 종결 (`~함.` `~됨.` `~음.` `~임.`) — 헤딩 / ASCII / 코드 주석 외엔 금지
- 학술 톤 종결 (`~한다.` `~이다.` `~없다.` `~된다.`) — 인용 / conditional 외엔 금지
- 명령조 (`~하라` `~할 것`) — 사용자 행동 권유는 `~하세요`

**판단 흐름**:
1. *이 문장이 규칙 / 정의 / 단정 / 정책 / 객관적 결과 인가* → 합쇼체
2. *이 문장이 독자에게 풀어 설명 / 대화 / 비유 / 흐름 인가* → 해요체
3. *이 문장이 헤딩 / 라벨 / 코드 주석 / ASCII 라벨 인가* → 명사구 OK
4. *이 문장이 인용 (큰따옴표 / "...라는 점") 또는 conditional (~한다면) 인가* → 학술 톤 허용

### 섹션명 자연 한국어

학술적 섹션명 대신 물음/감탄 형태 허용:

| ❌ 학술체 | ✅ 자연체 |
|---|---|
| 한 문장 직관 | 결론부터 |
| Context — 이 결정이 답해야 했던 물음 | 왜 이런 고민이 시작됐나? |
| Options Considered | 고민했던 대안들 |
| Consequences | 이 선택이 가져온 것 |
| Lessons Learned | 교훈 |

**예외**: 고유명사는 영문 유지 — `Status`, `Prior Art`, `Code References`, `ArchUnit`, `ADR`.

### 독자에게 직접 말 걸기 허용

- "~해요" 보다 직접적인 톤 OK: "지금은 쓸 데가 없어요", "처음 보는 사람에게는 혼란 가능"
- 감탄사 절제 — `ㅋㅋ`, 이모지 금지 (유저 요청 시만 예외)

### 표현 강도

- 단정적 표현 허용: "**전혀** 하지 않습니다", "**반드시** 따릅니다"
- 단 근거가 없으면 쓰지 말 것. 모든 단정은 뒤에 "왜" 를 수반해야 해요.
- 완곡 표현 (`~인 것 같습니다`, `~할 수도 있습니다`) 은 정말 불확실할 때만.

### 샘플

- 해요체 일관: [`ADR-012 · 앱별 독립 유저 모델 (통합 계정 폐기)`](../philosophy/adr-012-per-app-user-model.md) 전체

## 4. 용어집 (Glossary)

프로젝트 특유 용어는 **표기를 고정**. 흔들리면 검색/치환이 안 돼요.

| 용어 | 표기 | 설명 |
|---|---|---|
| 템플릿 레포 | `template-spring` | 본 레포 (GitHub Template Repository) |
| 파생 레포 | `server-factory`, `sumtally-backend` 등 | 템플릿에서 "Use this template" 으로 분기된 레포 |
| 앱 모듈 | `apps/app-<slug>` | 하나의 서비스 도메인을 담는 모듈 |
| core 모듈 | `core/core-*-api`, `core/core-*-impl` | 공통 도메인 (auth/user/device 등) |
| common 모듈 | `common/common-*` | 도메인 없는 공통 인프라 |
| Port | `AuthPort`, `UserPort` 등 | `-api` 모듈의 인터페이스 |
| Primary Adapter | Controller | HTTP → 서비스 방향 |
| Secondary Adapter | `ResendEmailAdapter` 등 | 서비스 → 외부 방향 |
| 솔로 | 한 명의 개발자 | "솔로 감당 가능" 등 |
| 도그푸딩 | `tools/dogfooding/` | 자체 템플릿을 자기 프로젝트에 적용하는 검증 |
| cherry-pick 전파 | — | 템플릿 레포 개선 → 파생 레포로 전파 |
| `appSlug` | 단일 문자열 claim | JWT 의 앱 식별자 ([`ADR-012`](../philosophy/adr-012-per-app-user-model.md)) |
| template-v* 태그 | `template-v0.3.0` 등 | 템플릿 레포 전체 단위 SemVer |

### 용어 사용 규칙

- 한 번 정의하면 본문에서 계속 같은 표기 유지
- 약어는 **첫 등장 시 풀어쓰기** + 괄호로 약어: `ADR (Architecture Decision Record)`
- 같은 개념을 한국어/영어 혼용 지양 — "포트" vs "Port" 중 하나로 고정 (본 프로젝트는 **Port** 영문 유지)

### 어려운 용어 cross-link 정책 (Level 0~1 reader 친화)

본 template 의 docs 는 *N개 파생 레포 / N명 개발자* 가 학습 자료로 사용해요. *처음 보는 사람* 의 학습 곡선 부담을 줄이기 위해:

- **L0~L1 진입 docs** (`onboarding/*`, `start/*`) — 어려운 용어 *첫 등장 시* `[term](../reference/glossary.md#section-anchor)` 형태로 cross-link **의무**
- **L2~L3 docs** (`structure/*`, `convention/*`, `philosophy/adr-*`) — 어려운 용어 *첫 등장 시* cross-link **권장** (Level 3 reader 가정이라 너무 많은 link 는 가독성 저해 — 핵심 *외부 도메인 용어* 만)
- **glossary 미커버 용어** — 발견 즉시 `docs/reference/glossary.md` 에 추가. 새 용어가 첫 등장하면 reader 가 막힘
- **반복 등장 시 link 생략** — 같은 ADR 안에서 같은 용어가 여러 번 등장하면 첫 1회만 link

**어려운 용어** 의 기준:
- 외부 도메인 용어 (Spring 관용 / Hexagonal / TOTP / IAP / RTDN / Kamal 등)
- 본 레포 고유 용어 (appSlug / 도그푸딩 / template-v* 태그)
- 첫 등장 시 비유 / 정의가 본문에 없는 경우

참조: [`first-change.md`](../onboarding/first-change.md) 의 *필수 선이해* 박스 — L0 진입 docs 의 cross-link 사례

## 5. 코드 블록 규칙

### 언어 태깅 필수

```java   ← ✅
```
```      ← ❌ (언어 미지정)
```

모든 코드 블록에 언어 태그 (`java`, `bash`, `yaml`, `sql`, `gradle`, `markdown` 등).

### 경로 주석

긴 코드 스니펫에는 **파일 경로 주석** 을 첫 줄에 달기. Truncation 여부도 명시.

```java
// common-security/jwt/JwtService.java 발췌
public String issueAccessToken(...) {
    ...
}
```

`발췌` 표기로 "전체가 아님" 을 명확히. 전체 인용이면 `전체`.

### 공백 / 들여쓰기

- 4 스페이스 (Java/Gradle 기본)
- 2 스페이스 (JSON/YAML/Markdown)
- 탭 금지

### 주석 최소

- 코드 스니펫 안의 주석은 **문서 맥락을 위한 것만**. 실제 코드의 주석이 아닌 것은 `// ← 이 부분이 핵심` 같은 지시 주석 허용.
- 단 이런 지시 주석은 스니펫 1~2개 이하로만. 많으면 본문에서 설명.

### 샘플

- 경로 주석 + 발췌 표기: [`ADR-006 · HS256 JWT (대칭키)`](../philosophy/adr-006-hs256-jwt.md) 의 JwtService 스니펫
- 지시 주석 활용: [`ADR-014 · Delegation mock 테스트 금지`](../philosophy/adr-014-no-delegation-mock.md) 의 "// ← 구현 세부에 결합" 주석

## 11. 형식 분기 가이드 — List ↔ Table ↔ Flow ↔ Prose

Level 3 ADR 의 정신은 *책의 형태로 풀어 쓴 깊이 있는 산문* 입니다. 모든 정보를 짧은 bullet 으로 쪼개 늘어놓는 *chatGPT 풍 열거형* 은 형식상 채운 인상을 주고 줄바꿈만 늘립니다. 정보의 성격에 맞는 형식을 고르는 게 핵심이에요.

모범으로 삼을 세 문서가 있어요. [`philosophy/README.md`](../philosophy/README.md) 의 *프롤로그* (제약 1/2/3 의 산문 흐름), [`ADR-001 · 모듈러 모놀리스`](../philosophy/adr-001-modular-monolith.md), [`ADR-002 · Use this template`](../philosophy/adr-002-use-this-template.md) 의 서두 — 셋 다 *큰 흐름은 2~3 단락 산문으로 풀어 쓰고, bullet 은 동일 레벨 짧은 항목 나열에만 쓰는* 톤입니다. 비유 (복권 사기 모델), 근거, 반례가 자연스럽게 섞여 들어가요.

### 11.1 Prose (산문) — Level 3 ADR 의 기본값

ADR 의 *결론부터*, *왜 이런 고민이 시작됐나*, *결정의 근거*, *이 선택이 가져온 것*, *교훈* 같은 섹션은 prose 가 기본입니다. 정보가 *전제 → 추론 → 결론* 의 연결된 흐름으로 흘러야 의미가 전달되기 때문이에요. bullet 으로 쪼개면 맥락이 단절되고 reader 가 *왜 이게 중요한가* 를 스스로 재구성해야 합니다.

예를 들어 *HS256 채택 근거* 를 다음처럼 bullet 으로 쪼개면 chatGPT 가 형식 채운 인상을 줘요.

```markdown
❌ bullet 으로 쪼개진 경우
- HS256 은 대칭키
- 발급자와 검증자가 같음
- JWKS 불필요
- 키 관리 환경변수 1개로 충분
```

같은 정보를 산문으로 풀면 *왜 이 알고리즘이 솔로 운영에 정합한가* 가 자연스럽게 드러납니다.

```markdown
✅ 산문 (자연)
HS256 은 대칭키 알고리즘이라 발급자와 검증자가 같은 비밀 키를 공유합니다.
그래서 JWKS 같은 공개키 배포 인프라가 필요 없고, 키 관리는 환경변수
`JWT_SECRET` 한 개로 끝나요. 솔로 운영의 *제약 2 — 시간 희소성* 에 정합하는
선택입니다.
```

산문이 적합한 자리는 *논리 전개*, *근거 설명*, *비유 / 예시*, *회고 / 교훈* 입니다. ADR 의 본문 대부분이 여기에 해당해요.

### 11.2 Table — 대안 비교 / 매트릭스 / 매핑

여러 항목이 같은 속성을 갖는 경우엔 표가 압도적으로 자연스러워요. 옵션 A/B/C 의 장점/단점/탈락이유를 비교하거나, 이벤트별 처리 매핑을 정리하거나, 컨텍스트별 분기 기준을 보여주는 자리가 그 예입니다. bullet 5개 이상이 같은 *콜론 패턴* (`- 항목: 값1, 값2, 값3`) 으로 늘어지면 거의 항상 표로 변환했을 때 가독성이 좋아져요.

```markdown
❌ list 형태 (어색)
- DID_RENEW: push 알림, "구독이 갱신됐습니다", 한국어
- REFUND: push 알림, "환불이 처리됐습니다", 한국어
- REVOKE: push 알림, "구독이 취소됐습니다", 한국어
```

```markdown
✅ table 형태 (자연)
| 이벤트 | 채널 | 메시지 | 언어 |
|---|---|---|---|
| DID_RENEW | push | "구독이 갱신됐습니다" | 한국어 |
| REFUND | push | "환불이 처리됐습니다" | 한국어 |
| REVOKE | push | "구독이 취소됐습니다" | 한국어 |
```

[`ADR-005`](../philosophy/adr-005-db-schema-isolation.md) 의 *대안 비교 표* 와 [`ADR-019`](../philosophy/adr-019-billing-iap-payment-separation.md) 의 *PG 비교 표* 가 그 모범이에요.

### 11.3 Flow diagram — 호출 흐름 / 상태 전이 / Phase 분리

순서가 정보의 핵심인 경우엔 ASCII flow 또는 mermaid 가 적합합니다. webhook 도착부터 markProcessed 까지의 phase 분리, ACTIVE → CANCELLED → EXPIRED 같은 상태 전이, A 가 B 를 부르고 B 가 C 를 부르는 호출 chain 이 그 예예요. bullet 안에 화살표 (`→`) 가 자주 등장하면 flow 후보로 검토할 신호입니다.

```
webhook 도착
   ↓
HMAC SHA-256 검증
   ↓
(source, externalId) UNIQUE 체크
   ↓
PaymentRecord INSERT (Phase 1, TX)
   ↓
PortOne API verify (Phase 2, NO TX)
   ↓
markProcessed (Phase 3, TX)
```

ADR 의 *결론부터* / *결정* 섹션이나 Explanation 의 *구조* 섹션이 ASCII 가 가장 잘 어울리는 자리예요. [`ADR-020`](../philosophy/adr-020-subscription-domain-model.md) 의 *결론부터* 흐름도와 [`ADR-022`](../philosophy/adr-022-iap-server-notifications.md) 의 *처리 매핑* 흐름도가 모범입니다.

### 11.4 Bullet list — 동일 레벨 짧은 항목 나열에만

bullet 이 적합한 자리는 한정적이에요. *동일 레벨의 짧은 평행 항목 나열*, *비순차 옵션 (3개 이하 권장)*, *본문 prose 의 간결한 부연*, *순서 있는 실행 명령 (이 경우 numbered)* 정도. 그 외에는 거의 모두 prose / table / flow 중 하나가 더 자연스러워요.

특히 *Level 3 ADR* 에서 다음 안티패턴은 *형식상 채운 인상* 의 주범이에요. 5개 이상의 bullet 이 같은 패턴으로 늘어지는 경우 (table 후보), bullet 안에 화살표가 자주 등장하는 경우 (flow 후보), bullet 들이 의미적으로 한 흐름을 이루는 경우 (prose 후보), 헤딩 다음 도입 prose 없이 바로 bullet 이 시작되는 경우, 표 행이 1~2개뿐인 경우.

### 11.5 형식 결정 흐름

새 정보를 적기 시작할 때 다음 순서로 판단하세요. 가장 많은 자리가 *prose* 라는 게 핵심입니다.

```
정보를 적기 시작
   │
   ├─ [항목 × 속성 매트릭스] 또는 [대안 비교] 인가? → table
   │
   ├─ [순서 / 흐름 / 상태 전이] 가 정보의 핵심인가? → ASCII flow / mermaid
   │
   ├─ [동일 레벨 짧은 항목 나열] 인가? (3~5개 이하 권장) → bullet
   │
   └─ 그 외 모든 [논리 전개 / 근거 / 비유 / 회고] → prose ★ Level 3 기본값
```

이미 작성된 문서를 리뷰할 때는 역방향으로 점검하세요. bullet 5개+ 가 같은 패턴이면 table 후보, bullet 안 화살표 다수면 flow 후보, bullet 들이 한 흐름이면 prose 후보로 변환을 검토합니다.
